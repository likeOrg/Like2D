### TSDoc coverage

Currently, TSDoc comments are sparse. Before generating docs, we need:
- [ ] `@module` tags on all core files
- [ ] `@example` blocks on public API methods
- [ ] `@param` / `@returns` on all public functions
- [ ] `@remarks` for design decisions and gotchas
- [ ] `@public` / `@internal` visibility tags

### Documentation pain points to address (v1.8.1)

Based on v2.8.0 planning, these concepts need clear explanation:

- **Scene lifecycle** - When does `load` vs `update` vs `draw` fire? What's the order?
- **Event binding scope** - Difference between canvas element vs window events, when to use each
- **Focus/blur behavior** - What triggers these, what the game should do in response
- **Input mapping** - How to bind actions, why use it over direct input checks
- **preventDefault strategy** - Which keys are blocked, how to customize, accessibility tradeoffs
- **Pointer lock** - When it activates, how to control it, mouse visibility modes
- **Scaling modes** - `setMode` options, pixel-perfect vs smooth scaling, fullscreen behavior

### IDE autocomplete priorities

Users want tooltips on:
- `Scene` interface methods (load, update, draw, event handlers)
- `Like` interface properties (gfx, audio, input, timer, etc.)
- `setMode` parameters and return types
- Input mapping API (`map`, `unmap`, `isDown`, `justPressed`)
- Graphics methods (`draw`, `clear`, `newImage`)
