# V3 Research Plan

## Reference

The Love2D main wiki (`love2d.org/wiki`) returns 403. Use the community API mirror instead:
- **https://love2d-community.github.io/love-api/** тАФ full Love2D 11.5 reference (modules, callbacks, types, config)

## Guiding Principle

Like2D is not a Love2D port. We share the same design space тАФ 2D game development with simple APIs тАФ but we solve problems our own way. We don't add features because Love2D has them. We add features because they solve real problems well, in our style.

When comparing to Love2D, the question is never "do we have this?" It's "what problem does this solve, and what's the right answer for us?"

## 1. Canvas API Audit

CanvasRenderingContext2D has ~45 methods and ~25 properties. Like2D wraps a subset. For each unwrapped feature, we decide: wrap it, leave it to `ctx` access, or reject it.

### Currently Wrapped

| Canvas2D API | Like2D Wrapper | Notes |
|---|---|---|
| `fillRect` / `strokeRect` | `graphics.rectangle(mode, color, rect)` | |
| `arc` + `fill`/`stroke` | `graphics.circle(...)`, `graphics.arc(...)` | circle is arc with ellipse scaling |
| `moveTo` / `lineTo` / `stroke` | `graphics.line(color, points)` | |
| `beginPath` / `moveTo` / `lineTo` / `closePath` / `fill`/`stroke` | `graphics.polygon(mode, color, points)` | |
| `fillRect(x,y,1,1)` per point | `graphics.points(color, points)` | |
| `fillText` | `graphics.print(color, text, position)` | |
| `drawImage` | `graphics.draw(handle, position, props)` | |
| `clearRect` (via fillRect with bg) | `graphics.clear()` | |
| `clip` via `beginPath`/`rect`/`clip` | `graphics.clip(rect?)` | |
| `createCanvas` (offscreen) | `graphics.newCanvas(size)`, `graphics.setCanvas()` | |
| `font` | `graphics.setFont(size, font?)`, `graphics.getFont()` | |
| `lineWidth`, `lineCap`, `lineJoin`, `miterLimit` | Passed via `ShapeProps` | Stateless per-call |
| `save` / `restore` | Used internally by `circle`, `draw` | Not exposed |

### Not Wrapped тАФ Research Required

**Shapes:**
- `ellipse()` тАФ Canvas has it, we don't. Is it worth a wrapper, or is `circle()` with non-uniform radii sufficient? Our `circle()` already supports `Vector2` radii.
- `roundRect()` тАФ in the Canvas spec but not Love2D. Useful for UI. Wrap or ctx?
- `bezierCurveTo()` / `quadraticCurveTo()` тАФ curves. Should we expose a path-building API or leave to ctx?
- `arcTo()` тАФ useful for rounded corners. Wrap or ctx?

**Paths:**
- `beginPath`, `closePath`, `moveTo`, `lineTo` тАФ we handle polyline via `line` and closed shapes via `polygon`. But no freeform path API. Should we add one?
- `Path2D` тАФ reusable path objects. Wrap or ctx?

**Text:**
- `strokeText()` тАФ outlined text. Uncommon. Skip?
- `measureText()` / `TextMetrics` тАФ needed for layout. Currently `print` does wrapping internally but doesn't expose metrics. Should we expose `graphics.measureText(text)`?
- `textAlign`, `textBaseline` тАФ currently hardcoded. Should these be props on `print()`?

**Gradients & Patterns:**
- `createLinearGradient()` / `createRadialGradient()` / `createConicGradient()` тАФ powerful but complex to wrap. Reject or expose as "advanced" API?
- `createPattern()` тАФ tiling fills. Useful for backgrounds. Wrap or ctx?

**Shadows:**
- `shadowBlur`, `shadowColor`, `shadowOffsetX`, `shadowOffsetY` тАФ Canvas feature, not a game engine staple. Better done with offscreen canvases. Reject?

**Compositing:**
- `globalAlpha` тАФ simple to wrap as a prop or method.
- `globalCompositeOperation` тАФ blend modes. Very useful for lighting, masking, etc. Wrap?

**Transforms:**
- `rotate`, `scale`, `translate`, `setTransform`, `resetTransform`, `getTransform` тАФ removed in V2 because users should access ctx directly. Do we stand by this, or was it premature?
- `transform()` (multiply matrix) тАФ advanced. Skip?

**Filters:**
- `filter` тАФ CSS filters on canvas (blur, brightness, etc.). Modern and powerful. No precedent in game frameworks. Wrap or ctx?

**Pixel Manipulation:**
- `getImageData` / `putImageData` / `createImageData` тАФ useful for screenshots, procedural generation, visual effects. Wrap?
- `imageSmoothingEnabled` / `imageSmoothingQuality` тАФ partially handled by pixel art mode. Should be exposed for `draw()`.

**State:**
- `save()` / `restore()` тАФ currently internal only. Should users call these for complex scenes?

### Research Questions
- Which features are "give me the chainsaw" territory (user calls ctx directly)?
- Which features would benefit from our stateless props-table pattern?
- Which features are rarely used and not worth the surface area?
- Does adding transforms back contradict V2's removal decision?

## 2. Audio API Audit

### Currently Wrapped

| Web Audio / HTMLAudio | Like2D Wrapper | Notes |
|---|---|---|
| `HTMLAudioElement` playback | `Source.play()`, `stop()`, `pause()`, `resume()` | Fire-and-forget deferred loading |
| `currentTime` | `Source.seek()`, `Source.tell()` | |
| `volume` | `Source.setVolume()`, `Audio.setVolume()` | Global volume scaling |
| `duration` | `Source.getDuration()` | |
| `paused` / `ended` | `Source.isPlaying()`, `isPaused()`, `isStopped()` | |
| Clone element | `Audio.clone(source)` | |

### Not Wrapped тАФ Research Required

**Pitch / Rate:**
- `playbackRate` on HTMLAudioElement тАФ trivial to add. Common need for variety in repeated sounds. Wrap?

**Looping:**
- `loop` on HTMLAudioElement тАФ currently requires `source.audio.loop = true`. Should we add `Source.setLooping(bool)`?

**Web Audio API (not HTMLAudioElement):**
- `AudioContext` / `AudioBuffer` тАФ we use HTMLAudioElement per source. Web Audio API enables:
  - Precise timing / scheduling
  - Spatial audio (panning, 3D position)
  - Real-time effects (reverb, delay, filters)
  - Audio analysis (FFT for visualizers)
  - Zero-latency playback
- **Question:** Migrate to Web Audio? Keep HTMLAudio for simplicity and add Web Audio as an advanced channel? Or stay out of real-time audio entirely?

**SoundData / Streamable Sources:**
- We only have streaming (HTMLAudioElement). No way to generate audio procedurally. Is this a gap worth filling?

**Recording:**
- `MediaRecorder` / `getUserMedia` тАФ mic input. Useful for some game types. Reject?

### Research Questions
- Is HTMLAudioElement "good enough" for games, or does it hit limits in practice?
- Would Web Audio API integration break the "lightweight" philosophy?
- Can we add pitch/looping to HTMLAudioElement without Web Audio?

## 3. Love2D Coverage тАФ Design Space Comparison

Love2D has been solving the same problems for 15+ years. Its feature set is a useful map of the problem space тАФ not a spec to match. For each area Love2D covers, we ask: does this problem exist on the web? How do we want to handle it?

### Areas We've Solved

| Problem Space | Love2D Approach | Like2D Approach |
|---|---|---|
| 2D shapes & text | `love.graphics` | `Graphics` class, stateless props |
| Audio playback | `love.audio` | `Audio` / `Source`, fire-and-forget loading |
| Keyboard input | `love.keyboard` | `Keyboard` class |
| Mouse input | `love.mouse` | `Mouse` class, with coordinate transforms |
| Gamepad input | `love.joystick` / `love.gamepad` | `Gamepad` class, SDL mapping database |
| Timing | `love.timer` | `Timer` class |
| Event dispatch | `love.event` | `Like2DEvent` discriminated union |
| Window/canvas sizing | `love.window.setMode` | `setScaling()` with fixed/native modes |
| Font configuration | `love.graphics.newFont` / `setFont` | `setFont(size, family)` тАФ simpler, web-native |

### Areas Worth Investigating

| Problem Space | Love2D Approach | Our Status | Priority |
|---|---|---|---|
| Blend modes | `love.graphics.setBlendMode` | Not wrapped | High |
| Touch input | `love.touch` | Not implemented | High |
| Sprite animation | Third-party (anim8) | Not implemented | Medium |
| Camera / viewport | Third-party (gamera, hump.camera) | Not implemented | Medium |
| Collision detection | Third-party (bump.lua) | Not implemented | Low |
| Math utilities | `love.math` (noise, random, bezier) | Partial (Vec2/Rect) | Medium |
| Pixel data / screenshots | `love.image` / `ImageData` | Not wrapped | Medium |
| Text measurement | Via `TextMetrics` | Internal only | Medium |
| Low-level audio | `love.sound` / `love.data` | Not implemented | Low |
| File I/O | `love.filesystem` | N/A (web has Fetch) | Low |
| Physics | `love.physics` (Box2D) | N/A (too heavy for core) | Low |
| Threading | `love.thread` | Web Workers тАФ complex | Low |
| System info | `love.system` | Partial (navigator APIs) | Low |

### Areas We've Deliberately Rejected

| Problem Space | Why Not |
|---|---|
| `setColor` stateful API | We pass color per-call. Stateful color is a source of bugs. |
| Window title/icon | Browser controls this, not us. |
| Built-in physics | Box2D is too heavy. Recommend a library instead. |
| BMFont / advanced text | Over-engineered for most games. Keep it simple. |

## 4. Community Solutions Survey

Love2D has a rich ecosystem of third-party libraries. These tell us what problems the core framework doesn't solve тАФ and what game developers actually need. We survey these to understand the demand landscape, not to copy them.

| Library | Problem It Solves | Like2D Status | Notes |
|---|---|---|---|
| **anim8** | Sprite animation from sprite sheets | Missing | Very common need. Sprite sheets + quads. |
| **gamera** / **hump.camera** | Camera with viewport, scrolling, zoom | Missing | V2 TODO flagged this for research. |
| **bump.lua** | AABB collision detection | Missing | Lightweight. Could be core. |
| **STI** | Tiled map rendering (TMX/JSON) | N/A | Tiled is a moving target. Guide instead of core. |
| **hump** (gamestate) | Scene/state management | Solved | We have SceneRunner. |
| **hump** (signal) | Event system | Solved | We have Like2DEvent. |
| **hump** (class) | OOP helpers | N/A | TypeScript has classes. |
| **lume** | Utility functions (lerp, clamp, map) | Partial | Vec2 covers vector math. Could extend. |
| **knife** | Utilities (serial, tween, etc.) | Partial | Consider individual pieces. |
| **suit** / **SUIT** | Immediate-mode GUI | N/A | Debug overlays only. Not core. |
| **Boon** | Input binding | Solved | We have action mapping. |
| **Profile** | Profiling / debugging | Missing | Useful but not high priority. |

### Research Questions
- Is sprite animation (anim8-like) the #1 most-missed feature from Love2D?
- Should collision detection (bump.lua-like) be core or a recommended library?
- Does camera (gamera-like) belong in core given the V2 TODO interest?

### Open Design Questions

2. Should the callback adapter's `like` object be a class with methods instead of an object with function properties?
3. Is the scene adapter's `handleEvent` тЖТ direct method dispatch the right pattern, or should it be one or the other?
4. Should `ImageHandle` and `Source` share a common `AssetHandle` interface?
